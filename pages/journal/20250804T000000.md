clear vs compact 차이:

clear:

-   대화 내용을 완전히 삭제
-   깨끗한 새 시작
-   이전 대화 내용 접근 불가

compact:

-   대화 내용을 AI가 요약해서 압축
-   핵심 정보는 유지하되 길이만 줄임
-   요약된 형태로 이전 맥락 보존

현재 상황에서는 clear가 더 적합합니다:

-   모든 핵심 정보가 이미 문서 파일에 저장됨
-   다음 작업이 명확함 (Inspector 개선)
-   긴 대화 기록이 집중력을 방해함

compact를 쓸 때:

-   대화 맥락을 일부 유지하고 싶을 때
-   중간 결과물이 문서로 정리되지 않았을 때

@assistant LiteLLM은 주로 LLM API를 통합하고 관리하는 백엔드 프록시 서버 역할을 합니다. 자체적으로 OpenWebUI와 같은 완벽한 프론트엔드를 제공하지는 않습니다.

하지만 LiteLLM을 사용하는 개발자들이 프론트엔드를 구축할 때 다음과 같은 방법을 고려할 수 있습니다:

-   **OpenAI 호환 API**: LiteLLM은 OpenAI API 형식을 따르므로, OpenAI API와 호환되는 기존 프론트엔드 라이브러리나 애플리케이션을 LiteLLM 서버에 연결하여 사용할 수 있습니다.
-   **커스텀 프론트엔드 개발**: 조직의 특정 요구사항에 맞춰 React, Vue, Angular 등 웹 프레임워크를 사용하여 커스텀 프론트엔드를 개발하고, 이 프론트엔드에서 LiteLLM 서버로 요청을 보내는 방식입니다.
-   **기존 LLM 프론트엔드 솔루션 활용**: OpenWebUI, Chatbot UI 등 오픈 소스 LLM 프론트엔드 솔루션을 LiteLLM과 연동하여 사용하는 방법도 있습니다. 이 경우, 해당 프론트엔드의 설정에서 API 엔드포인트를 LiteLLM 서버 주소로 변경해야 합니다.

LiteLLM 공식 문서나 GitHub 저장소에서는 특정 프론트엔드를 "추천"하기보다는, 개발자가 유연하게 선택할 수 있도록 API 호환성에 중점을 두고 있습니다.
