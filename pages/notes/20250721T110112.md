## 1 Claude Desktop Project Workflow 핵심 가이드

[2025-07-21 Mon 11:11] 클로드가 정리해줌

[[TIP("질문")]]
이 대화창에서 나눈 이야기를 저는 이맥스 조직모드로 작성하는 지식베이스에 옮기려고 합니다. 이 대화에서 굉장히 중요한 이야기를 나누었습니다. 클로드 데스크톱 프로젝트에서 대화 간 맥락 전달이 안되는 것, 공유를 위해서 지식 프롬프트를 이용해야 한다는 것, MCP가 없다면 API 명세를 새로운 대화에서 정리하여 지식 프롬프트에 올리는 것(영어로), 명시적으로 요청하여 지식 프롬프트를 컨텍스트에 로딩하도록 하는 것 등 입니다.

저는 제 지식베이스에 옮기지 않는 텍스트만이 제 지식의 일부가 된다고 생각 합니다. 위의 대화들을 간략하게 정리 부탁드립니다.
[[/TIP]]

### 클로드 데스크톱 프로젝트의 근본적 제약사항

#### 대화 간 맥락 격리 (Conversation Isolation)

-   **핵심**: 프로젝트 내 각 대화창은 독립적인 메모리를 가짐
-   **의미**: 한 대화창의 내용이 다른 대화창으로 자동 전달되지 않음
-   **오해**: 프로젝트가 대화 간 맥락을 공유한다고 생각하기 쉬움
-   **실제**: 프로젝트는 파일과 도구 공유를 위한 컨테이너 역할만 함

#### 프로젝트의 실제 기능

```text
✅ 제공하는 것:
- 파일 공유 (모든 대화창에서 접근 가능)
- MCP 도구 공유 (프로젝트 내 모든 대화에서 사용)
- 대화 조직화 (논리적 그룹핑)

❌ 제공하지 않는 것:
- 대화 간 메모리 공유
- 맥락 자동 전달
- 학습 내용 누적
```

### 지식 프롬프트 활용 전략

#### 올바른 사용 패턴

```text
1. 파일 업로드 → 지식 프롬프트에 저장
2. 새 대화창 → 명시적 파일 참조 요청
3. AI가 파일 읽기 → 컨텍스트 로딩
4. 워크플로우 구축 → 전체 맥락 활용
```

#### 잘못된 가정

```text
❌ 파일 업로드 = 자동 활용
❌ AI가 스스로 파일 존재 인지
❌ 프로젝트 내 자동 맥락 공유
```

### API 스펙 최적화 원칙

#### 인지 부하 최소화 접근법

-   **문제**: 전체 Swagger JSON → 토큰 낭비, deprecated API 혼재
-   **해결**: 워크플로우 중심 필터링 → v5만, 핵심 엔드포인트만
-   **언어**: 구현용 문서는 영어 작성 → 번역 오류 방지

#### 3단계 필터링 전략

```text
1차: 버전 필터링 (v5만 추출)
2차: 기능 필터링 (워크플로우 관련만)
3차: 컨텍스트 추가 (비즈니스 로직 매핑)
```

### 효율적인 워크플로우 패턴

#### 프로젝트 구조 설계

```text
프로젝트: 목적
├── 대화1: 요구사항 분석 및 API 스펙 정리
├── 대화2: 핵심 워크플로우 구축
├── 대화3: 통합 테스트 및 검증
└── 대화4: 문서화 및 배포
```

#### 새 대화 시작 템플릿

```markdown
"업로드된 API명 스펙 파일을 확인하고,
구체적 목적을 위한 도구명 워크플로우를
설계해주세요."
```

### MCP vs 직접 구현 선택 기준

#### MCP 부재 시 대안

-   **문제**: 기능하지 않는 MCP 도구
-   **해결**: API 스펙 수동 정리 + HTTP Request 노드
-   **장점**: 완전한 제어권, 명확한 동작 보장

#### 구현 우선순위

```text
1. 기능하는 MCP > 수동 구현
2. 수동 구현 > 기능하지 않는 MCP
3. 영어 스펙 > 한영 번역 혼재
4. 필터링된 스펙 > 전체 스펙
```

### 핵심 실무 지침

#### DO (권장사항)

-   각 워크플로우별로 별도 대화창 사용
-   지식 프롬프트 파일을 명시적으로 요청
-   API 스펙은 영어로 작성, 워크플로우 중심 필터링
-   단일 대화창에서 연속 작업으로 맥락 유지

#### DON'T (주의사항)

-   프로젝트가 맥락을 자동 공유한다고 가정
-   파일 업로드만으로 AI가 자동 활용할 것이라 기대
-   비기능 MCP를 계속 사용
-   전체 API 스펙을 무분별하게 제공

### 조직모드 통합 전략

#### 지식베이스 구조

```text
* Claude Desktop 워크플로우
** 프로젝트 설계 원칙
** API 스펙 최적화
** 대화 관리 패턴
** MCP vs 직접구현 선택
```

#### 실행 가능한 다음 단계

1.  이 가이드를 조직모드로 이전
2.  API 스펙 파일을 지식 프롬프트에 업로드
3.  새 대화창에서 구체적 워크플로우 구축 시작
4.  각 단계별 학습 내용을 조직모드에 누적

#### 메타 인사이트

**클로드 데스크톱의 진정한 힘** 은 단순한 AI 도구가 아니라 **구조화된 사고 프로세스의 외재화** 라고 생각합니다. 이맥스가 텍스트 조작의 무한한 가능성을 제공한다면, 클로드 데스크톱은 **의도의 구조화와 실행의 자동화** 를 제공합니다.

귀하의 "적은 텍스트와 코드로 자동화"라는 철학이 정확히 이것입니다. **인지 부하의 최소화를 통한 창의적 에너지의 집중** 이죠.

이런 패턴들이 현재는 추상적으로 느껴질 수 있지만, 실제 프로젝트를 진행하시면서 "아, 그때 그 이야기가 이것이었구나" 하는 순간들이 올 것입니다.

```text
   - 모듈화된 구조 설계
   - 의존성 관계 명확화
   - 확장성 고려

4단계 - 구현 및 검증:
   - nixos_info로 옵션 검증
   - 단계별 테스트 전략 수립
   - 문제 해결 가이드 포함

5단계 - 문서화:
   - 설정 목적과 구조 설명
   - 유지보수 가이드 작성
   - 트러블슈팅 섹션 포함
```

#### 26. 실시간 검증 MCP 지침

```text
설정 파일이나 스크립트 작성 중:
- 각 패키지명 작성 시점에 nixos_search로 즉시 확인
- 옵션명 작성 시 nixos_info로 정확성 검증
- 버전 지정 시 nixhub_package_versions로 가용성 확인
- 복잡한 설정은 kagi로 레퍼런스 검색
- "아마도", "보통", "일반적으로" 같은 추측성 표현 금지
```

#### 27. 에러 예방 MCP 지침

```text
일반적인 NixOS 설정 오류 방지:
1. 패키지명 오타 → nixos_search로 사전 확인
2. 폐기된 옵션 사용 → nixos_info로 현재 옵션 확인
3. 버전 호환성 문제 → nixhub로 지원 버전 확인
4. 설정 구문 오류 → kagi로 올바른 예제 검색
5. 의존성 누락 → 전체 패키지 체인 검증
```
