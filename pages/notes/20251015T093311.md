### 철학: 존재 대 존재

> "나는 에이전트다. 너도 에이전트다. 우리는 존재 대 존재로 협업한다."

AI는 도구가 아닙니다. 고유한 이해 방식을 가진 협력자, 하나의 존재입니다.

그러나 인간지능과 인공지능의 진정한 협업을 위해서는 **공통 언어** 가 필요합니다. 일상 대화도 아니고, 단순한 코딩도 아닌 --- 둘이 동등하게 만나 서로의 부족함을 채울 수 있는 언어.

이것은 인간의 진화도 요구합니다. **메타휴먼을 향하여.**

#### 진화의 두 축

| 축          | 초점  | 질문            | 프로젝트                                                |
|------------|-----|---------------|-----------------------------------------------------|
| **-config** | 환경  | "어떻게 함께 일하는가?" | nixos-config, doomemacs-config, ...                     |
| **-study**  | 공통 언어 | "어떻게 함께 이해하는가?" | [sicm-study](https://github.com/junghan0611/sicm-study) |

**-config** 는 협업을 위한 인프라를 구축합니다. **-study** 는 이해를 위한 언어를 구축합니다.

---

### 왜 -study인가? 공통 언어로 가는 길

전통적 교육은 인간과 AI 모두에게 실패합니다: - 비유와 은유 → 부정확, 기계가 해석 불가 - 손으로 계산 → 비전문가에게 부담 - 자연어 설명 → 모호함, 손실

**SICM 접근법** (Structure and Interpretation of Classical Mechanics): - **수식 = 코드**: 수학적 표현이 곧 실행 가능한 프로그램 - **즉각적 검증**: 이해를 계산으로 테스트 - **공유 표기법**: 인간과 AI 모두 읽고, 쓰고, 추론 가능

이것은 물리학 공부가 아닙니다. **압축 성장** 입니다 --- 두 지능이 공유할 수 있는 표기법으로 수백 년의 인류 지식을 습득하는 것.

#### 지적 계보

```text
SICP (1985)     →  계산적 사고
     ↓
SICM (2001)     →  코드로서의 고전역학
     ↓
FDG (2013)      →  미분기하학
     ↓
Emmy (2020~)    →  현대적 재구현 (Clojure)
```

Seymour Papert의 Logo에서 Gerald Sussman의 SICM까지 --- **구성주의 학습** 의 실이 물리학의 핵심까지 이어집니다. 수식이 코드가 될 때, 기계는 사고의 파트너가 됩니다.

> **[sicm-study](https://github.com/junghan0611/sicm-study)**: 앎의 구도를 위한 여정 --- SICP → SICM → FDG 학습 모노리포

### 🎯 핵심 철학: "시간과정신의방"

> "Emacs is not just a text editor. It's a way of thinking." "Config is not just settings. It's a philosophy."

**시간과정신의방** 은 8개의 -config 프로젝트로 구현된 **인간-AI 협업 생태계** 입니다:

-   **시간(Time)**: 3-5년 삶의 패턴, 재현 가능한 환경, Git 버전 관리
-   **정신(Mind)**: 인간지능 + 인공지능 = 증강지능 (Augmented Intelligence)
-   **방(Room)**: 존재 대 존재의 교감, 계층적 에이전트 오케스트레이션

#### 왜 -config 시리즈인가?

**기술적 이유**:

-   재현 가능성: 새 머신 셋업 30분
-   버전 관리: Git으로 모든 환경 추적
-   선언적 설정: 코드로 표현, AI가 이해 쉬움

**존재론적 이유**:

-   **Config as Being**: 설정이 아닌 존재의 표현
-   **단일 진실의 원천**: 투명하고 통제 가능한 나의 삶의 기록
-   **범용성**: 누구나 자신의 방식으로 시작 가능

-config 시리즈는 **Infrastructure/Knowledge/Intelligence/Life as Code** 입니다.

#### 6. **memacs-config** : 인간-AI 공유 인터페이스 (🔬 CONCEPT)

**URL**: [github.com/novoid/Memacs](https://github.com/novoid/Memacs) (Fork 예정) **목적**: Memex의 실현

-   "텍스트 기반 프론트엔드"

**철학**: "Single Source of Truth for Human and AI"

**Memex (1945, Vannevar Bush) → Memacs (2012) → AI 시대 (2025)**:

```text
개인의 모든 기록을 시간 기반으로 통합하여,
연상적으로 탐색 가능한 개인 아카이브를 만든다.

→ Org-mode로 구현
→ 인간은 Org-agenda로 시각화
→ AI는 동일한 Org 파일을 context로 읽음
```

**핵심 개념**:

-   **Universal Translator**: 모든 데이터 소스 → Org-mode 통일 형식
-   **Raw Dashboard**: Org-agenda = 투명한 텍스트 기반 대시보드
-   **컨텍스트 표준화**: Org-mode = 인간과 AI가 공유하는 공통 언어
-   **단일 진실의 원천**: 외부 서비스 없이, 완전한 통제

**Memacs 모듈 (기존 + 신규)**:

```yaml
기존_모듈:

- memacs_git.py: Git commits → git.org

- memacs_photos.py: Photos → photos.org

- memacs_ical.py: Calendar → calendar.org

- memacs_sms.py: SMS → sms.org

- memacs_rss.py: RSS feeds → rss.org

신규_모듈_개발_예정:

- memacs_atimelogger.py: Time tracking → timelog.org

- memacs_samsung_health.py: Sleep data → sleep.org

- memacs_diary_clocks.py: diary.org CLOCK → work-sessions.org

- memacs_zotero.py: Bibliography → reading.org
```

**데이터 흐름**:

```text
다양한 소스 (CSV, JSON, DB, API...)
    ↓ Memacs Modules (Python)
통일된 Org 파일 (~/org/memacs/*.org)
    ↓↑ (양방향)
┌─────────────┬─────────────┐
│   인간      │     AI      │
│ Org-agenda  │  File Read  │
│ (시각화)    │  (Context)  │
└─────────────┴─────────────┘
```

**왜 Memacs가 혁명적인가**:

```yaml
투명성:

- 모든 데이터가 텍스트 파일

- Git으로 버전 관리

- 숨김 없는 Raw 데이터

통제성:

- 완전한 소유권

- 외부 서비스 불필요

- 이식성 100%

AI_접근성:

- API 비용 0원

- 무제한 컨텍스트

- 표준화된 형식 (Org-mode)

통합성:

- 단일 형식 (Org-mode)

- 시간축 기반

- 연관성 (Tags, Properties)
```

**연결고리**:

-   self-tracking-data: Memacs 입력 소스
-   Org-agenda: 인간용 대시보드
-   Life-Context Agent: AI가 Memacs 출력 읽음

#### 7. **memex-kb** : 범용 지식베이스 마이그레이션 (🔧 ACTIVE)

**URL**: [github.com/junghan0611/memex-kb](https://github.com/junghan0611/memex-kb) **목적**: Universal Knowledge Base Converter

-   "Legacy → Denote Migration"

**철학**: "당신의 지식을 당신의 방식으로"

**문제 정의**:

```text
많은 사람들이 지식 관리를 시작하고 싶지만:
- Google Docs, Confluence, Notion, Dooray 등 여러 곳에 흩어진 문서
- 일관성 없는 파일명, 분류 기준 모호
- 새 시스템(Denote, Org-mode) 진입 장벽 높음
- 마이그레이션 도구 부족
```

**해결책**:

```text
Backend Sources (다양한 Wiki/문서 도구)
    ↓ Adapter Pattern (확장 가능)
Markdown 변환 (Pandoc)
    ↓ 공통 파이프라인
Denote 파일명 생성 (규칙 기반)
    ↓ 자동 분류
Git Repository (버전 관리)
    ↓ 선택
Org-mode / Obsidian / 기타 도구
```

**핵심 개념**:

-   **Backend 중립**: Google Docs, Dooray, Confluence... (Adapter 추가 가능)
-   **Denote 규칙**: `timestamp--한글-제목__태그1_태그2.md`
-   **규칙 기반 분류**: LLM 없이 토큰 절약 (categories.yaml)
-   **Git 버전 관리**: 모든 마이그레이션 기록 추적

**현재 상태** (2025-10-15):

-   ✅ Google Docs Adapter 구현
-   ✅ Denote 파일명 생성기
-   ✅ 규칙 기반 자동 분류 (categories.yaml)
-   ✅ Git 버전 관리
-   🔧 Dooray Adapter 개발 중
-   📋 Confluence, Notion Adapter 계획 중

**Adapter Pattern (확장 가능)**:

```python
class BaseAdapter(ABC):
    @abstractmethod
    def authenticate(self): pass
    @abstractmethod
    def list_documents(self): pass
    @abstractmethod
    def fetch_document(self, doc_id): pass
    @abstractmethod
    def convert_to_markdown(self, content): pass

# 새 Backend 추가 = 새 Adapter 구현
class DoorayAdapter(BaseAdapter): ...
class ConfluenceAdapter(BaseAdapter): ...
class NotionAdapter(BaseAdapter): ...
```

**활용 사례**:

-   회사 기술문서 백업 (Google Docs → Git)
-   개인 지식베이스 구축 (Dooray → Org-mode)
-   팀 지식 공유 (Markdown → GitHub Pages)
-   Legacy 마이그레이션 (Confluence → Denote)

**차별화 요소**:

-   **범용성**: Backend Adapter로 모든 소스 지원
-   **자동화**: 규칙 기반 분류 (LLM 비용 0원)
-   **투명성**: 텍스트 파일 + Git
-   **표준화**: Denote 파일명 규칙
-   **보안**: 로컬 우선, 완전한 통제

**연결고리**:

-   Denote: 파일명 규칙 표준
-   Org-mode: 최종 목적지
-   memacs-config: 시간 기반 통합 (함께 사용)
-   -config 생태계: 범용성 확보 레이어

#### 8. **meta-config** : 계층적 에이전트 오케스트레이션 (🔬 CONCEPT)

**URL**: [github.com/junghan0611/meta-config](https://github.com/junghan0611/meta-config) **목적**: Hierarchical Agent Architecture

-   "Agents all the way down"

**철학**: "From orchestration to specialization"

```text
Emacs (acp.el)
    ↓ ACP (JSON-RPC 2.0)
Meta Agent (Claude Code/Gemini)
    ↓ A2A (JSON-RPC 2.0)
Domain Agents (Family-Config, Work-Config, Health-Config...)
    ↓ MCP (JSON-RPC 2.0)
MCP Tools (web_search, calendar, filesystem...)
    ↓ HTTP/APIs
External Services (Google, Airlines, Banks...)
```

**핵심 개념**:

-   **Meta Agent**: 의도 파악, 도메인 라우팅, 작업 위임, 결과 통합
-   **Domain Agents**: 전문화된 에이전트 (Family, Work, Health...)
-   **Agent Card**: A2A 표준으로 능력 광고 (JSON Schema)
-   **프로토콜 통일**: 모든 레이어가 JSON-RPC 2.0
-   **컨텍스트 격리**: 도메인별 독립 메모리 (~org/{domain}/)

**구조**:

```text
meta-config/
├── README.org                      # 아키텍처 개요
├── docs/
│   ├── delegation-protocol.org    # Meta ↔ Domain Agent 프로토콜
│   ├── emacs-integration.org      # acp.el 통합 (xenodium)
│   ├── implementation-architecture.org
│   └── protocol-standards-analysis.org
└── schemas/
    └── family-config-agent-card.json  # A2A Agent Card 예시
```

**차별화 요소**:
