## Emacs AI 도구 비교: Wrapper vs CLIProxyAPI vs 직접 API

[2026-02-25 Wed] Reddit r/emacs 스레드 ([LLMs and agents in Emacs](https://www.reddit.com/r/emacs/comments/1r78p48/)) 에서 ahyatt(llm.el), karthink(gptel), xenodium(agent-shell) 대화 분석 정리.

### 문제: Claude 구독 vs API 과금

| 방식             | 비용                  | 용도                   |
|----------------|---------------------|----------------------|
| Claude API (토큰당) | 사용량 비례 (3~15 달러/MTok) | 프로덕션, 정밀 제어    |
| Claude Pro 구독  | 월 20 달러 (한도 있음) | 가벼운 개인 사용       |
| Claude Max 구독  | 월 100~200 달러       | 파워 유저 (Claude Code 포함) |
| OpenRouter       | 사용량 비례 (다양)    | 멀티 모델              |

파워 유저는 API 과금이 월 50~200 달러 쉽게 넘는다. 구독을 API처럼 쓰면 절약 가능.

### 세 가지 접근법 비교

#### 방법 1: claude-code-openai-wrapper (현재 사용 중)

```nil
[gptel] → HTTP → [FastAPI :8000] → [Claude Agent SDK] → [Claude Code] → [Anthropic]
                  (Python 서버)       (subprocess)
```

-   **장점**: 도구 사용(tool-use) 완전 지원 — Read, Write, Bash, WebSearch 등
-   **장점**: OpenAI API 호환 → gptel에서 백엔드로 바로 사용
-   **단점**: 서버 별도 실행 필요 (`./run.sh`)
-   **단점**: 레이턴시 ~10-50초 (다단계 프록시 + Claude Code 초기화)
-   **단점**: Python + FastAPI + 12개 모듈 → 무거움
-   **ToS**: ⚠️ 회색지대 (Claude Code SDK 래핑)
-   **적합**: 코딩 에이전트 작업 (파일 편집, 코드 생성, 웹검색)

#### 방법 2: CLIProxyAPI (Go 바이너리 프록시) — 추천 우회로

```nil
[gptel] → HTTP → [CLIProxyAPI :8317] → [OAuth Token] → [Anthropic API]
                  (Go 바이너리)          (직접 호출)
```

-   **장점**: 레이턴시 1~3초 (직접 API 호출)
-   **장점**: 싱글 Go 바이너리 — 런타임 의존성 없음
-   **장점**: systemd 서비스로 항상 실행 가능
-   **장점**: Claude + Gemini + OpenAI Codex 멀티 프로바이더
-   **장점**: 멀티 계정 라운드로빈 지원
-   **단점**: 도구 사용(tool-use) 불가 — 채팅만
-   **단점**: OAuth 토큰 주기적 갱신 필요
-   **ToS**: ❌ 위반 (2026.02.20 Anthropic ToS 명시적 금지)
-   **적합**: 빠른 질의응답, 코드 리뷰, 번역, 요약

참고: [rogs 블로그](https://rogs.me/2026/02/use-your-claude-max-subscription-as-an-api-with-cliproxyapi/) — Emacs llm 패키지 + CLIProxyAPI 연동 가이드. forge-llm, magit-gptcommit 등에서 사용.

#### 방법 3: OpenRouter API (현재 기본)

```nil
[gptel] → HTTPS → [OpenRouter] → [각 프로바이더]
                   (API 키)
```

-   **장점**: 합법, 안정적, 멀티 모델 (Gemini, GPT, Claude)
-   **장점**: 레이턴시 0.5~2.5초
-   **단점**: 토큰당 과금
-   **적합**: 일상적 사용, 모델 비교, 프로덕션

### 이중 전략 (권장)

| 작업 유형          | 도구             | 백엔드                     |
|----------------|----------------|-------------------------|
| 빠른 질문 _번역_ 요약 | gptel            | CLIProxyAPI (Claude-Proxy) |
| 코드 리뷰 /커밋 메시지 | gptel/magit      | CLIProxyAPI                |
| 코딩 에이전트 (파일 편집) | pi / Claude Code | Claude Code 직접           |
| 멀티 모델 비교     | gptel            | OpenRouter                 |
| Gemini Flash (초고속) | gptel            | OpenRouter                 |

핵심: **단순 채팅은 CLIProxyAPI (1~3초), 에이전트 작업은 pi/Claude Code 직접**. Wrapper(50초)를 채팅용으로 쓰는 건 과도함.

### ToS 현황 (2026.02 기준)

[The Register (2026.02.20)](https://www.theregister.com/2026/02/20/anthropic_clarifies_ban_third_party_claude_access): Anthropic이 ToS를 명시적으로 업데이트함:

> "Using OAuth tokens obtained through Claude Free, Pro, or Max accounts in any other product, tool, or service — including the Agent SDK — is not permitted and constitutes a violation of the Consumer Terms of Service."

-   모든 서드파티 하니스(pi, OpenCode, Cline 등) 구독 사용이 공식 금지
-   OpenCode는 Anthropic 법적 요청으로 Claude 구독 지원 코드를 [제거함](https://github.com/anomalyco/opencode/commit/973715f3da1839ef2eba62d4140fe7441d539411)
-   반면 OpenAI는 Codex 구독의 서드파티 하니스 사용을 [공식 허용](https://x.com/thsottiaux/status/2009742187484065881)
-   karthink(gptel 저자)도 "ToS 이유로 공식 배포 안 함, 사용자가 직접 설정"이라는 입장

**실용적 판단**: 개인 로컬 사용은 현실적으로 단속 어려움. 단, 계정 정지 리스크 인지 필요. .gitignore 처리하여 공개하지 않을 것.
