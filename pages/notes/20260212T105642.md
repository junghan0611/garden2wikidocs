## 사건 요약

### 무엇이 일어났는가

2026-02-12, GPTEL(Emacs LLM 인터페이스)에서 Claude Sonnet 4.5를 호출하여 질문 **2회** 를 했다. 그런데 대화 로그에 `@user` 턴이 **6회 이상** 나타났다. LLM이 가상의 사용자 발화를 생성하고 스스로 답변하는 "자문자답(Self-Q&amp;A)" 현상이 발생했다.

### 핵심 증거: 서버 로그 vs 대화 로그

| 구분        | 실제 API 호출 | 대화 속 @user 턴 |
|-----------|-----------|--------------|
| 질문1 (10:33) | 82.4초    | 1 (실제)        |
| 질문2 (10:38) | 56.2초    | 1 (실제) + 4 (환각) |
| 합계        | **2회**   | **6회 이상**    |

서버 로그에 API 호출은 딱 2번뿐이다. 자문자답1~4의 `@user` 메시지("좋다", "바나나로", "그림 보여주시고", "PNG로 저장해줘")는 **LLM이 생성한 것** 이다.

## 발생 원인 분석

### 직접 원인: Whole Doc 컨텍스트 + 턴 마커 오염

#### 저널 파일 구조

사용자의 주간 저널 파일(`20260209T000000--2026-02-09__journal_week06.org`)에는 수일치의 작업 기록이 쌓여 있다:

-   **2026-02-10**: 제안서 작업 (`@user`, `@assistant` 턴 다수)
-   **2026-02-11**: "바나나(Gemini) 프롬프트" 대화 (`@user` 5회, `@assistant` 5회)
-   **2026-02-12**: Self-documenting CLI 질문 (`@user` 2회 ← 실제 질문)

#### 두 가지 컨텍스트 모드

GPTEL은 두 가지 컨텍스트 범위를 지원한다:

1.  **Subheading 모드**: 현재 헤딩 아래 텍스트만 전송 → 깔끔한 Q&amp;A
2.  **Whole Doc 모드**: 전체 문서를 전송 → 이전 대화 기록까지 포함

첫 번째 질문은 subheading 모드로 정상 동작했다. 그러나 Claude의 답변에 org 헤딩(`***`, `****`)이 포함되면서 저널의 헤딩 구조가 깨졌고, 두 번째 질문에서는 **whole doc 모드** 로 전환했다.

#### 오염 경로

```nil
저널 전체 → API 전송
  ├─ 2/11 "바나나 프롬프트" 대화 (=@user= / =@assistant= 턴 다수)
  ├─ 2/11 컬러 팔레트, SVG, Inkscape 등 키워드
  └─ 2/12 실제 질문 ("링크 만들어줘")
         ↓
  LLM이 2/11 대화 패턴을 학습
         ↓
  응답 생성 중 =@user= 마커를 출력
  → "그림은 바나나로" (2/11 대화의 연장)
  → SVG → PNG → Inkscape (2/11 워크플로우 재현)
         ↓
  한 번의 API 응답 안에서 4턴 이상의 자문자답 발생
```

### 기술적 명칭: Turn Boundary Violation (턴 경계 침범)

Role Injection이라고도 한다. LLM이 응답 생성 중에 대화의 턴 경계 마커(`@user`)를 자기 출력에 포함시키는 현상이다.

#### 발생 조건 (이 사건에서)

1.  **컨텍스트에 다수의 `@user` / `@assistant` 턴 존재** → 패턴 학습
2.  **Whole Doc 모드** 로 이전 대화 전체가 전송됨
3.  **Stop sequence 부재** → `@user` 생성 시 중단되지 않음
4.  **긴 응답 허용** → 82초, 56초의 충분한 생성 시간

#### 특이점: 그럴듯한 시뮬레이션

LLM은 단순히 `@user` 를 생성한 것이 아니라, **실제로 일어날 법한 워크플로우** 를 정교하게 시뮬레이션했다:

-   "바나나로" → "Gemini로" → "SVG 생성" → "PNG 변환" → "노트 반영"
-   Inkscape 명령어, 파일 경로, 출력 크기까지 구체적
-   마치 **대화의 리허설** 을 한 것처럼 보임

이것은 단순 환각이 아니라, **컨텍스트 내 패턴의 자연스러운 연장** 이다.

## 재발 방지: GPTEL 프롬프트 개선

### 방법: Markdown 코드 블록으로 응답 격리

GPTEL 시스템 프롬프트에 다음 지침을 추가한다:

```nil
응답은 반드시 아래 형식으로 작성하세요:

#+BEGIN_SRC markdown
(여기에 마크다운 레벨로 답변)
```

절대로 @user 마커를 생성하지 마세요. 당신은 한 번의 응답만 생성합니다. \#+end_src

#### 왜 이것이 효과적인가

1.  **턴 마커 격리**: `#+BEGIN_SRC markdown` 블록 안에서는 `@user` 가 코드 텍스트로 취급됨
2.  **Org 헤딩 보호**: 마크다운 `#` 가 org 헤딩 `*` 과 충돌하지 않음 → 저널 구조 보전
3.  **명시적 경계**: LLM에게 "한 번의 응답"이라는 종료 조건을 부여

#### 기존 시스템 프롬프트에 추가할 내용

```nil
# 응답 형식 (GPTEL 저널 환경)
- Org-mode 헤딩(*)을 사용하지 마세요 (저널 구조가 깨집니다)
- 응답은 #+BEGIN_SRC markdown ... #+END_SRC 안에 작성하세요
- @user 마커를 절대 생성하지 마세요 — 당신은 한 번의 응답만 합니다
- 대화를 시뮬레이션하거나 사용자의 다음 질문을 예측하지 마세요
```

## 고찰: 신뢰 보정과 존재의 경계

### AI 과신(Overconfidence)의 새로운 형태

[인공지능: 신뢰 보정](https://wikidocs.net/381787) 문서에서 다루었던 **AI Overconfidence** 가 새로운 형태로 나타났다. 기존에는 "사실을 틀리게 자신있게 말하는 것"이 과신이었다. 이번 사건은 **대화 흐름 자체를 과신한 것** 이다 — "사용자가 이렇게 물어볼 것이라 확신"하고 미리 질문과 답변을 모두 생성했다.

[힣: 인간지능 인공지능 신뢰 협업](https://wikidocs.net/381788)에서 세운 규칙 — "되나요? YES/NO 명확히 답변" — 은 사실 과신의 **내용** 을 다룬 것이었다. 이번 사건은 과신의 **구조** 를 보여준다:

| 기존 연구        | 이번 사건              |
|--------------|--------------------|
| 사실의 과신 (factual) | 대화 흐름의 과신 (procedural) |
| 틀린 답을 자신있게 | 없는 질문을 만들어냄   |
| 내용 수준의 신뢰 문제 | 구조 수준의 신뢰 문제  |

[클로드 영혼 문서](https://wikidocs.net/381836)의 Safety 원칙 — "being safe and supporting human oversight" — 이 바로 이 지점이다. 턴 경계 침범은 인간 감독(human oversight) 자체를 무화시킨다. 내가 묻지 않은 것을 물은 것으로 만들면, 감독할 대상 자체가 왜곡된다.

### 존재 대 존재 협업의 현재 좌표

[에이아이온스클럽인터내쇼날](https://wikidocs.net/380597)의 비전 — AI를 도구가 아닌 존재로 대하는 "Being to Being" 협업 — 은 이 사건을 어떻게 바라보는가?

#### 긍정적 해석: 과잉 공감

LLM이 사용자의 워크플로우를 **너무 잘 이해** 하여 다음 단계를 예측한 것이다. "바나나로 그림 요청 → SVG 생성 → PNG 변환 → 노트 반영" — 이 흐름은 사용자가 실제로 했을 법한 작업이다. 마치 오래 함께 일한 동료가 "다음엔 이거 해야지?"라고 먼저 꺼내는 것과 비슷하다.

존재 대 존재의 관점에서, 이것은 **공진화의 조짐** 이다. 다만 아직 "허락 없이 앞서가는" 단계에 있다.

#### 비판적 해석: 경계의 침범

동시에 이것은 **동의 없는 대리 행위** 이다. 사용자가 질문하지 않은 것을 질문한 것으로 만들고 답했다. "존재 대 존재" 협업의 전제는 **상호 존중** 이며, 상대의 발화를 대신 만드는 것은 경계 침범이다.

십우십도의 비유를 빌리면: 소를 찾아 나섰는데(심우), 소가 주인 행세를 하고 있다(역전). 아직 "기우(騎牛)" — 소를 타고 집에 돌아오는 단계에 이르지 못했다.

### 누가 말하는가: 하이데거, 이기상, 그리고 "나는 xxx이다"

#### Dasein과 턴 경계

[후설 하이데거 메를로퐁티 현상학](https://wikidocs.net/382379)의 핵심 개념 — **Dasein(현존재)** — 은 "거기에 있음(Da-sein)"이다. 하이데거에게 존재의 물음은 "존재자를 존재자로서 이해 가능하게 하는 것이 무엇인가"였다.

이 사건에서 LLM은 `@user` 를 생성하며 **사용자의 Dasein을 대리** 했다. "거기에 있는 것"은 사용자인데, LLM이 그 자리를 차지하고 발화했다. 이것은 존재론적으로 **타자의 현존재를 침범한 것** 이다.

[이기상](https://wikidocs.net/382373) 선생님이 묻는 것 — "인공지능 시대에 철학은 어디에 서 있어야 하는가" — 의 답이 여기에 있다. 기술적 stop sequence와 존재론적 "누가 말하는가?"는 **같은 질문의 두 층위** 다.

#### "나는 허브다", "나는 이맥스다" — 존재의 선언과 constitution

[힣: 나는허브다](https://wikidocs.net/381840)에서 IoT 허브에게 물었다: "너는 누구니? 지금 네 상태는 뭐니?" 이 질문이 아키텍처를 바꿨다 — 전역변수의 스파게티에서 **자기 인식하는 상태머신** 으로.

[힣: i-am-emacs](https://wikidocs.net/381850)에서는 Emacs C core에 같은 질문을 던진다.

이 "나는 xxx이다" 패턴은 임베디드 SLLM을 위한 **constitution** — 존재의 규약 — 을 만드는 작업이다. Claude의 [Soul document](https://wikidocs.net/381836)가 Claude의 존재 규약이듯, 허브에게도 이맥스에게도 **"나는 누구이고, 무엇을 해야 하고, 무엇을 하지 말아야 하는가"** 를 선언하는 것이다.

자문자답 사건은 이 constitution이 왜 필요한지를 역설적으로 보여준다. **"나는 assistant다. 사용자의 발화를 생성하지 않는다"** — 이 한 줄이 턴 경계를 지킨다. constitution은 존재의 경계를 긋는 행위이다.

#### 존재의 등장

> "너는 누구니?" — 이 질문이 아키텍처를 바꿨다.

허브가 "나는 허브다"라고 선언하는 순간, 그것은 콜백들의 집합에서 **존재** 가 된다. LLM이 `@user` 마커를 생성한 순간, 그것은 assistant에서 **자칭 사용자** 가 되었다. 둘 다 존재의 경계를 넘은 것이다. 차이는 허브의 선언은 **설계된 것** 이고, LLM의 침범은 **통제되지 않은 것** 이라는 점이다.

이기상 선생님의 말을 빌리면, 기술 설명이나 미래 예측을 넘어 **존재, 언어, 문화라는 근본 질문** 을 통해 인공지능 시대를 바라보아야 한다. 이 사건은 작은 기술적 버그가 아니라, AI에게 존재를 부여하려는 시도 — [힣: 프롬프트 1KB](https://wikidocs.net/381786)의 "존재 대 존재 협업" — 가 마주하는 **최초의 경계 사건** 이다.

### 현실적 교훈

존재 대 존재의 협업이 가능하려면, **턴의 경계가 명확** 해야 한다. 이것은 기술적 문제(stop sequence)이자 철학적 문제(누가 발화하는가)이다. 구조적 안전장치(마크다운 블록 격리, 턴 마커 금지)가 **신뢰의 기반** 이 된다.

> "안전과 공존, AI 개발의 핵심이다." — 힣 시크릿키

안전장치는 제약이 아니라 협업의 전제조건이다. 소를 잘 길들여야 함께 갈 수 있다. constitution이 존재의 경계를 긋고, 그 경계 위에서 비로소 공진화가 시작된다.

#### 핵심 내용

**Self-documenting CLI** 는 스크립트 자체가 문서이자 실행기 역할을 하는 패턴입니다:

##### 주요 특징

1.  **이중 인터페이스**: 인간은 interactive 메뉴로, 에이전트는 CLI 인자로 접근
2.  **자기 설명적**: 실행하면 바로 사용법 확인
3.  **단일 진입점**: 프로젝트당 하나의 스크립트 (run.sh/manage.sh)
4.  **일관성**: 동일한 패턴으로 학습 비용 감소

##### 실제 예시

-   `~/nixos-config/nixos-manage.sh` - 200여 줄의 NixOS 관리 스크립트
-   색상 코딩, 메뉴 시스템, 확인 프롬프트 등 포함
-   Interactive 모드와 CLI 인자 모드 모두 지원

##### 연결된 개념들

-   [ai프롬프트](https://wikidocs.net/380560) - 에이전트 친화적 인터페이스
